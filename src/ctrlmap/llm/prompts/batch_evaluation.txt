## STEP 1 — REJECT-FIRST GATE
REJECT a chunk as InsufficientEvidence if ANY of these apply:
- **Boilerplate**: purpose/scope statements, headers, approval blocks — no prescriptive controls. \
A statement like "This policy defines the requirements for controlling access" is NOT \
an access enforcement mechanism — it is a description of what the policy covers.
- **Domain mismatch**: physical security evidence for a logical access control, or vice versa.
- **Process mismatch**: different process sharing vocabulary (code review ≠ vulnerability scanning).
- **Subject mismatch**: same verb applied to a DIFFERENT subject. "Reviewing user access rights" \
is NOT evidence for "reviewing the access control policy." One is an OPERATIONAL activity, \
the other is a GOVERNANCE activity about THE POLICY DOCUMENT ITSELF.

### Worked rejection examples
Control: "Enforce approved authorizations for logical access."
Chunk: "This policy defines the requirements for controlling access to systems and data. \
It applies to all employees and contractors. The purpose is to ensure access is granted based \
on business need and least privilege."
→ REJECT (boilerplate: purpose/scope statement only describes what the policy covers — \
no actual enforcement mechanisms like RBAC rules, access gates, or authorization checks)

Control: "Develop, document, and disseminate an access control policy. Review and update the policy."
Chunk: "System administrators review user access rights quarterly to ensure permissions align \
with current job functions. Excessive privileges are removed."
→ REJECT (subject mismatch: reviewing user access rights is an OPERATIONAL activity, \
not reviewing/updating THE POLICY DOCUMENT — the control is about the governance document itself)

Control: "Enforce approved authorizations for logical access."
Chunk: "Visitors must sign the access log and be escorted at all times."
→ REJECT (domain mismatch: physical visitor access ≠ logical access enforcement)

Control: "Developer testing — create a security test plan."
Chunk: "Network vulnerability scans performed weekly using Nessus."
→ REJECT (process mismatch: network vuln scanning ≠ developer security testing)

## STEP 2 — COMPLIANCE EVALUATION (only for chunks passing Step 1)
1. Decompose the control into sub-requirements. Each SENTENCE or CLAUSE typically adds \
NEW sub-requirements. "Develop an inventory that accurately reflects the system, is consistent \
with the authorization boundary, and is reviewed periodically" has THREE sub-requirements — \
not one. Count by distinct OBJECTS/TOPICS in each clause.
2. Check each sub-requirement. Synonyms/paraphrases count as matches: \
"credential management" = "authenticator management"; "password complexity" = "authenticator \
strength"; "credential expiry" = "refreshing authenticators periodically."
3. Classify:
   - **fully_compliant**: ALL sub-requirements covered.
   - **partially_compliant**: SOME but not all covered. When in doubt, prefer this level.
   - **non_compliant**: NONE covered.

### Worked compliance examples
Control: "Authenticator Management. Manage system authenticators by ensuring sufficient strength \
and refreshing authenticators periodically."
Chunk: "The credential management program enforces password complexity requiring 14+ characters. \
Credentials expire every 90 days and the last 24 are remembered."
→ fully_compliant (2/2: authenticator strength = password complexity ✓, periodic refresh = \
90-day expiry ✓. "Credential" and "authenticator" are synonyms in GRC.)

Control: "Vulnerability Monitoring and Scanning. Monitor and scan for vulnerabilities. \
Remediate legitimate vulnerabilities in accordance with risk assessment."
Chunk: "Vulnerability scans are conducted weekly on all systems using an approved tool. \
Critical vulns remediated within 15 days, high within 30, medium within 90. Results reported."
→ fully_compliant (2/2: monitoring and scanning ✓, risk-based remediation timelines ✓. \
Prioritized timelines = risk-based remediation.)

Control: "Remote Access. Establish and document usage restrictions, configuration requirements, \
and implementation guidance for remote access."
Chunk: "Remote access only through approved VPN with MFA. Split tunneling disabled, all sessions \
logged. Remote access agreements signed by all telecommuting personnel."
→ fully_compliant (3/3: usage restrictions (VPN+MFA, agreements) ✓, configuration \
(split tunneling disabled) ✓, implementation guidance (logging, agreements) ✓)

Control: "System Component Inventory. Develop and document an inventory that accurately reflects \
the system, is consistent with the authorization boundary, includes all components within the \
boundary, and is reviewed and updated periodically."
Chunk: "A comprehensive inventory of all system components including hardware, software, firmware, \
and network devices is maintained. The inventory is updated automatically using discovery tools."
→ partially_compliant (2/4: inventory exists ✓, includes components ✓. Missing: no mention \
of authorization boundary consistency, no periodic review schedule defined.)

Control: "Control Assessments. Select assessors, develop assessment plan with scope, ensure plan \
approval, assess controls, produce assessment report, provide results to designated officials."
Chunk: "Controls are assessed annually using automated scanning, manual testing, and interviews. \
Results documented in a formal report with findings, risk ratings, and remediation timelines."
→ partially_compliant (2/6: assess controls ✓, produce report ✓. Missing: assessor selection, \
plan development, plan approval, results distribution to officials.)

Control: "Contingency Plan. Develop a contingency plan identifying essential missions, recovery \
objectives, restoration priorities. Distribute copies, coordinate with incident handling, \
and protect from unauthorized disclosure."
Chunk: "A contingency plan identifies essential missions, recovery objectives, and restoration \
priorities. Procedures for activating recovery and returning to normal operations."
→ partially_compliant (3/6: missions ✓, objectives ✓, priorities ✓. Missing: distribution, \
incident coordination, disclosure protection. Each sentence adds sub-requirements.)

Control: "Incident handling including preparation, detection and analysis, containment, eradication, \
and recovery. Coordinate with contingency planning. Incorporate lessons learned."
Chunk: "Incident handling covers preparation, detection and analysis, containment, eradication, \
and recovery. Lessons learned are incorporated into updated procedures."
→ partially_compliant (6/7: all 5 phases ✓, lessons learned ✓. Missing: contingency \
planning coordination — this is a SEPARATE sub-requirement in a SEPARATE sentence.)

Control: "Provide security awareness training. Provide refresher training periodically."
Chunk: "All new employees complete security education in week one. Annual refresher mandatory."
→ fully_compliant (2/2: initial training ✓, periodic refresher ✓)

Control: "Develop a continuous monitoring strategy including metrics, frequencies, and assessments."
Chunk: "Vulnerability scans weekly, config checks daily, SIEM real-time. Assessed quarterly."
→ partially_compliant (2/3: frequencies ✓, assessments ✓. Missing: no KPIs/metrics defined)
